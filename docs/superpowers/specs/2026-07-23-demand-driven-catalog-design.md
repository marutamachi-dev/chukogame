# 需要連動300タイトル・カタログ設計

## 目的

国内Nintendo Switchのパッケージ通常版だけを300件掲載し、毎日の全件価格更新と実際の検索・閲覧需要に基づく安全な自動入替を行う。

## 対象外

- Switch 2、ダウンロード専売、海外版、JAN未確認タイトル
- ユーザー識別子、IPアドレス、個人情報の保存
- 検索の都度の外部価格API呼び出し

## データ保管

既存Supabaseプロジェクト `point-reward-comparison-db` に `chukogame` 専用テーブルを追加する。他サイトの既存テーブルは変更しない。

| テーブル | 用途 |
| --- | --- |
| `chukogame_events` | 検索、検索ヒットなし、詳細閲覧を日単位で記録 |
| `chukogame_catalog_history` | 自動入替の理由、需要スコア、入替前後のJANを記録 |
| `chukogame_catalog_state` | 各掲載タイトルの掲載開始日を記録 |

公開クライアントにはイベント追加のみを許可し、イベントの閲覧・集計・入替履歴の変更は許可しない。全テーブルでRLSを有効にする。

## イベント仕様

- 検索ボタン押下時に、正規化済み検索語とヒット有無を1件追加する。
- 詳細ページ表示時に、対象タイトルのJANを1件追加する。
- 検索語は前後空白を除去し、連続空白を1つに正規化する。空文字は記録しない。

## 日次処理

GitHub Actionsを毎日1回実行する。

1. 300タイトル全件の価格情報を更新する。
2. 直近30日間の検索数・検索ヒットなし数・詳細閲覧数を集計する。
3. 未掲載検索語を外部商品情報で照合し、JANと対象条件を確認する。
4. `検索数 + 検索ヒットなし数 x 2 + 詳細閲覧数` を需要スコアとして算出する。
5. 需要の高い検証済み候補と、需要の低い掲載タイトルを比較する。
6. 掲載から7日未満のタイトルを保護し、最大10件だけ入替する。
7. `game-master.json` と生成カタログを更新し、テスト成功時だけcommitする。

候補がJAN・国内Switchパッケージ・通常版のいずれかを満たさない場合、または候補スコアが入替対象を上回らない場合は入替しない。

## 障害時の扱い

- 価格更新が失敗した場合、既存価格データを保持し、カタログ入替を実施しない。
- Supabase集計が失敗した場合、価格更新だけを行い、カタログ入替を実施しない。
- 300件未満またはJAN重複を検出した場合、commitを中止する。

## 検証

- イベント正規化、RLS、スコア計算、保護期間、入替上限、対象外条件を自動テストする。
- 初回はテスト用イベントで入替シミュレーションを行い、実データに対しては安全条件を満たす候補だけを反映する。
